00 · Start here

入口与共通原则

使用 Godot 4.7 打开工程并等待首次资源导入完成。轻量配置工具位于 Dock；地图、规则、单元格、UI、音频与角色动画使用顶部主屏，避免复杂工作流被挤在狭窄侧栏。

BTL 数值 右上 Dock Data_BalanceOverrides.json 新建模型、新开对局
BTL 规则 顶部主屏 Data_BalanceOverrides.json 验证并保存后，新建模型或新开对局
BTL 地图 顶部主屏 maps/Data_Map*.json 读取地图后传入新对局

先看黄色圆点。

工具标题出现「● 未保存」时，必须点击工具自己的保存按钮。只保存当前 Godot 场景，不等于保存这些编辑器数据。

五条不可破坏的数据规则

  • 默认数值和基准文案只存在于 src/Script_GameData.gd。
  • 编辑器写入覆盖层或数据文件，不直接修改权威默认表。
  • 运行时只读取当前模型的 model.tables 与 model.cfg，禁止修改 Rules.* 常量内容。
  • Data_ArtGeneration.json 只保存不可变原始提示词、追加修订、非敏感任务身份与派生资产证据；不保存凭据，也不被运行时读取。
  • 开始批量调整前先确认 Git 状态，保存后只审查对应数据文件的差异。

入口不见了怎么办

打开「Project → Project Settings → Plugins」，确认对应的 Btl*Editor 插件已启用。当前工程默认启用内容、数值、规则、地图、视效、UI、音频、单元格、抠图与角色动画十个插件。重新启用后仍未出现，先检查底部输出面板中的脚本错误，再重启 Godot。

00A · Agent interface

所有编辑器共用一套 Agent / CLI 契约

RunEditorTools.ps1 是十个编辑器数据面的统一入口。GUI 负责所见即所得、试听、图集点击与动画播放；Agent 负责可审计的读取、验证、dry-run、CAS 写入和状态查询，不再复制各个 Dock 或主屏的内部逻辑。

每次操作先发现能力。

Agent 在检查或修改编辑器管理的数据前，必须先执行 list，再对目标工具执行 describe。以返回的 action、参数、管理路径、交互边界和外部额度边界为准。

.\RunEditorTools.ps1 list -Json
.\RunEditorTools.ps1 describe rules -Json
.\RunEditorTools.ps1 rules show --table UT --entity militia -Json

cell_art映射 schema/show/set/clear/reset/validate；生成 assets/status/set_prompt/queue/dispatch/environment图集点击与实时 Map2D 预览只在 GUI；regenerate、confirm、dispatch 有额度边界

写操作的固定顺序

  1. 先读取并取得 sourceHash保存这次决策依据的精确版本。
  2. 先跑 --dry-run验证类型、引用、目标路径和变更内容，不写盘、不启动 worker。
  3. 带 --expected-hash 正式写入确保写回的仍是刚刚审阅的版本；退出码 4 表示 CAS 冲突，必须重新读取并重放。
  4. 解析 JSON 或结果前缀自动化优先使用 -Json；普通输出只读取最后一条 BTL_EDITOR_TOOL_RESULT 结果行，不能靠日志猜成功。
.\RunEditorTools.ps1 rules set --table UT --entity militia --field str --value 10 `
  --expected-hash <sourceHash> --dry-run -Json

# dry-run 通过后，复用同一 sourceHash 并移除 --dry-run 才会写入。

语义退出码

0 成功；2 用法错误；3 验证失败；4 CAS 冲突；5 I/O 或外部服务失败；6 部分完成。

Provider 额度红线

只有用户明确授权后，regenerate / confirm / dispatch 等外部成本动作才能带 --credit-confirmed。回归与文档验收不得消费真实额度，也不得自动替用户补确认参数。

01 · Localization

BTL 内容

把 GameData 中允许本地化的文本转成稳定键值，并在一个表格里维护各语言译文。键与数据路径一一对应，例如 UT.militia.name、WAR_PHASES.0.name。

入口右下「BTL 内容」Dock
数据Data_StringTable.json
基准语言zh-CN

添加或更新翻译

  1. 点击「从游戏数据提取」，扫描白名单数据表并补充尚不存在的词条。
  2. 在语言输入框填写 BCP-47 代码，例如 en、ja 或 zh-TW，点击「➕语言」。
  3. 从下拉框选择目标语言，双击「译文」列填写。清空译文会删除该语言值。
  4. 使用搜索框按键、基准文本或译文过滤；勾选「只看缺翻译」集中处理空项。
  5. 确认标题状态后点击「保存」，写入字符串表。
缺翻译会自动回落。

目标语言没有内容时，运行时显示基准语言文本，不会出现空白 UI。

提取按钮实际会做什么

  • 只新增缺失词条，或补上空缺的基准文本。
  • 不会覆盖已有基准文本，也不会覆盖已有翻译。
  • 如果上游基准文案有意修改，需要手工同步既有基准文本。
  • 新增了可本地化的数据表或字段时，还要更新 Script_StringTable.gd 的 TEXT_FIELDS 白名单。

运行时表现

字符串表包含两个及以上语言时，游戏 Debug 面板出现「语言」下拉框。切换语言会重建当前模型数据表、刷新界面并保存选择。保存后，已经运行的游戏不会自动监听文件变化；切换一次语言使其重读，或重新启动游戏。

当前覆盖范围有限。

只有 GameData 白名单中的数据表文案会被提取。Script_GameModel.gd 等脚本里的内联日志文本仍属于后续阶段。

02 · Balance

BTL 数值

用于调单位战斗参数、维护与造价，也可覆盖任意数值型 CFG。文件只记录与默认值的差异，因此平衡实验不会改动 DataParity 保护的权威表。

入口右上「BTL 数值」Dock
数据Data_BalanceOverrides.json
范围单位参数 + 数值 CFG
这是快速数值入口，不是完整规则数据库。

只改单位数值或少量 CFG 时用「BTL 数值」最快；需要跨表搜索、关系追踪、科技前置、类型与引用验证、ChangeSet 或 Agent 调用时，进入 BTL 规则编辑器。两者共享同一个覆盖文件，不要同时打开旧快照后交叉保存。

调整单位

  1. 在「单位参数」下拉框选择单位。
  2. 修改战力、移动力、视野、每回合维护，或粮食、人力、军火、组织造价。
  3. 黄色标签与「●」表示该项已有覆盖。点击单项右侧「↺」会删除覆盖并恢复默认。
  4. 点击「保存」写入覆盖文件，新开的对局才会使用新值。

添加 CFG 覆盖

  1. 在 CFG 搜索框输入键名的一部分，例如 charges。
  2. 从候选列表选中目标键，点击加号。
  3. 修改新出现的数值行；不再需要时点击该行「↺」。
刚加入的 CFG 也算覆盖。

即使数值尚未改变，它也会出现在覆盖文件中。不需要时主动点击「↺」，避免制造无意义差异。

顶部三个按钮

保存 / 全部重置

「保存」写入磁盘。「全部重置」只在内存中清空全部覆盖，还要再次保存才会形成空覆盖文件。

重新载入

放弃 Dock 中尚未保存的改动，从磁盘重新读取覆盖文件。

什么时候生效

  • 新建 GameModel 或新开对局时应用。
  • 已运行模型与进行中的存档保留原值，不会热更新。
  • 地图编辑器每次生成都会新建模型，因此会读取已经保存的覆盖。
  • MapParity 在剥离覆盖层的纯净默认数据上运行，普通平衡覆盖不会改变基准哈希。
03 · World generation

BTL 地图

直接调用游戏的 GameModel.GenMap，按固定种子和参数生成可复现地图。中央预览用于快速检查地形、村庄、铁路和设施布局。

入口顶部「BTL 地图」主屏
数据maps/Data_Map*.json
同源算法GameModel.GenMap

生成一张地图

  1. 填写种子；相同配置与相同种子应生成相同地图。
  2. 设置地图宽、高、村庄数、村庄最小间距和土匪山寨数。
  3. 点击「生成」。骰子按钮会随机更换种子并立即生成。
  4. 检查中央六边格预览，以及左侧各地形、村庄、铁路、炮楼、县城和匪寨统计。

预览色块和标记只用于布局验收，不代表游戏内最终 2D 或 3D 美术效果。

保存与命名

  1. 在顶部输入显示名，可以使用中文，也可以留空。
  2. 点击「保存」，文件写入 res://maps/。
  3. 文件主干会整理为 Data_Map<AsciiPascalCase>.json；没有可用 ASCII 字符时退回 Data_MapSeed<种子>.json。
  4. 完整显示名保存在 meta.name，不依赖文件名保存。
同名文件会直接覆盖。

地图文件只包含生成元数据、格子数据和总部位置，不包含单位、回合进度或其他进行中对局状态。

读回与开局

点击刷新按钮更新列表，选择 Data_Map* 后点击「读取」。这只会更新编辑器预览和参数显示，不会自动启动游戏。代码侧开局入口如下：

var loaded := TaihangMapFiles.LoadMap("Data_MapExample")
model.NewGame(int(loaded.meta.get("seed", 0)), loaded.map)

地图通常是体积较大的单行 JSON。人工检查时优先使用本工具的速览，或调用 TaihangMapFiles.MapStats()，不要依赖逐行阅读。

04 · Rendering

BTL 视效

在不启动完整游戏的情况下调整真正的 Script_Map3D 渲染路径。预览台包含四种地形、村庄、铁路、设施、单位与三级战争迷雾。

入口右上「BTL 视效」Dock
数据Data_VisualSettings.tres
预览Scene_VisualPreview.tscn

推荐流程

  1. 点击「预览场景」，打开 Scene_VisualPreview.tscn。
  2. 保持 3D 编辑器视口可见，在 Dock 中调整滑条、颜色、选项和开关。
  3. 预览满意后点击「保存」，写入视效 Resource。
  4. 如果游戏正在运行，在 Debug 面板点击「重载视效配置」拉取刚保存的结果。
分组可调整参数
相机俯角、FOV、镜头距离
环境系统时刻、时刻驱动氛围、体积雾、雾密度、天空、太阳、环境光
画质色调映射、曝光、TAA、性能模式
级联阴影关闭 / 2 级 / 4 级、阴影距离
单元格边缘（2.5D）白晕颜色与强度、噪声侵蚀、颗粒尺度、春夏秋冬边缘带颜色

时刻与手工颜色

关闭时刻驱动氛围

默认模式。时刻主要控制太阳角度，天空、太阳与环境光采用手工颜色。

开启时刻驱动氛围

昼夜曲线接管天空、太阳与环境光，手工颜色不再是最终显示值。

检查器与恢复默认

「检查器」会在 Godot Inspector 中打开同一个 Resource，适合精确输入。「恢复默认」只修改内存值并标记为未保存；确认采用默认值后仍须点击「保存」。Dock 修改会先作用于内存，因此预览可在保存前实时变化。

05 · Recipes

常用工作流与生产契约

AI 原始素材统一契约

凡是准备继续抠图、切片、打包或转成动画输入的 AI 原始素材图，单位、地块与散布物、UI 都使用同一条 provider 入口规则。

  • 每次请求恰好一张：一次 ImageGen、Lovart 或其他图像 provider 请求只接受一个可下载的源图 artifact，不接受多张备选图。固定网格的 atlas 或 UI kit 可以在这一张图内包含多个对象，仍只算一个源图 artifact。
  • 直接生成纯白底：provider 返回的源图必须使用不透明、均匀、纯 #FFFFFF 背景；提示词目标始终是精确白，文件门禁只容许 provider 导出量化造成的每个 RGB 通道最多 2 级偏差（253–255）。禁止透明背景、烘焙棋盘格、品红底、渐变、纹理、环境场景背景或本地洗白冒充原图。
  • 派生物另行处理：透明参考图、供 Dreamina 使用的 #FF00FF 输入、视频帧、最终资产与图集、接触表及其他 QA 证据都由源图之后的确定性流程产生，不反向充当 provider 原始素材图。

调一版平衡

  1. 在「BTL 数值」中只修改目标项并保存。
  2. 新开对局验证；涉及生成参数时，在「BTL 地图」中重新生成。
  3. 检查 Data_BalanceOverrides.json 的 Git 差异。
  4. 迭代期间跑快速回归，提交前跑全量回归。

增加一种语言

  1. 从游戏数据提取，添加语言代码并填写译文。
  2. 使用「只看缺翻译」处理全部空项并保存。
  3. 在游戏 Debug 面板切换语言，检查回落和界面刷新。
  4. 以 1920 × 1080 检查裁切、对齐、弹窗和模态边界。

制作并验收地图

  1. 记录种子和全部生成参数。
  2. 生成后用速览检查设施数量与地形分布。
  3. 使用明确的 ASCII 文件名部分和可读显示名保存。
  4. 重新读取一次，并在真实对局中验证可游玩性。

制作单位帧动画

从正确原画基准开始，经 ImageGen、Lovart 或其他图像 provider 生成一张纯白底源图，再通过 Dreamina、连续时间窗采样、抠图、固定脚线与游戏投影完成可追溯接入。

打开单位帧动画生产规范 →

完整规则与关系图

在 Default / Override / Effective 三层之间检查真实生效值，沿正反向引用评估影响，验证无误后再保存最小差异。

打开 BTL 规则编辑器手册 →

「BTL 单元格」检查 dual-grid 地表与邻接地貌合同；正式 Forest、Hills、Mountain 生产遵守独立的 12 图集 / 64-mask 规范。透明 PNG 诊断属于「BTL 抠图」。

06 · Troubleshooting

常见故障

Dock 或「BTL 地图」入口不见了

检查 Project Settings → Plugins，查看输出面板脚本错误，然后重启 Godot。

改动在重启后消失

检查工具标题是否仍显示「未保存」，使用工具自己的保存按钮。

数值在当前存档中没有变化

这是预期行为。保存后新开对局验证，进行中的存档不会被覆盖。

翻译保存后当前游戏没有变化

切换一次语言让模型重读字符串表，或重新启动游戏。

视效预览变化了，游戏却没有变化

先点击「保存」，再在运行时 Debug 面板点击「重载视效配置」。

地图列表没有刚保存的文件

点击右侧刷新按钮。仍缺失时检查输出面板和 maps/ 目录。

JSON 或 Resource 无法读取

检查最近的手工编辑是否破坏格式，优先从 Git 恢复对应数据文件。

规则编辑器拒绝保存

先查看底部诊断。Error 会阻断保存；如果提示文件已变化，说明磁盘版本与当前会话的 SHA-256 不一致，重新载入后再重放改动。

生成按钮灰掉或状态一直等待
